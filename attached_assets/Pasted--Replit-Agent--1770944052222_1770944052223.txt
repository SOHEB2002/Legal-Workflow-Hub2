# أمر Replit Agent التكميلي: تصنيف القضايا حسب الصفة وتعديل مسار العمل

## الهدف
حالياً النظام يفترض أن كل القضايا من نوع واحد (مدعي يرفع القضية من الصفر). لكن الواقع أن القضايا ثلاثة أنواع مختلفة في مسار العمل. المطلوب تعديل النظام ليدعم الأنواع الثلاثة مع مسار عمل ومراحل مختلفة لكل نوع.

---

## القسم 1: إضافة حقل تصنيف القضية

### 1.1 في `shared/schema.ts`:

أضف enum جديد لتصنيف القضية:

```typescript
export const CaseClassification = {
  PLAINTIFF_NEW: "مدعي_قضية_جديدة",
  PLAINTIFF_EXISTING: "مدعي_قضية_مقيدة",
  DEFENDANT: "مدعى_عليه",
} as const;

export type CaseClassificationValue = typeof CaseClassification[keyof typeof CaseClassification];

export const CaseClassificationLabels: Record<CaseClassificationValue, string> = {
  "مدعي_قضية_جديدة": "مدعي - قضية جديدة (نرفعها نحن)",
  "مدعي_قضية_مقيدة": "مدعي - قضية مقيدة (مرفوعة مسبقاً)",
  "مدعى_عليه": "مدعى عليه",
};
```

### 1.2 أضف حقل `caseClassification` في جدول `law_cases`:

```typescript
caseClassification: varchar("case_classification", { length: 50 }).notNull().default("مدعي_قضية_جديدة"),
```

### 1.3 أضف الحقل في `insertCaseSchema`:

```typescript
caseClassification: z.enum(["مدعي_قضية_جديدة", "مدعي_قضية_مقيدة", "مدعى_عليه"]),
```

### 1.4 أضف الحقل في interface `LawCase`:

```typescript
caseClassification: CaseClassificationValue;
```

شغّل `drizzle-kit push` لتحديث قاعدة البيانات.

## القسم 2: مراحل مختلفة لكل تصنيف

### 2.1 تعريف المراحل حسب التصنيف:

أضف في `shared/schema.ts`:

```typescript
// ==================== مراحل القضية حسب التصنيف ====================

// النوع 1: مدعي - قضية جديدة (نرفعها من الصفر)
export const PlaintiffNewStages: CaseStageValue[] = [
  "استلام",
  "استكمال_البيانات",
  "دراسة",
  "تحرير_المذكرة",         // هنا: تحرير صحيفة الدعوى
  "إحالة_للجنة_المراجعة",
  "الأخذ_بالملاحظات",
  "رفع_للدائرة",           // رفع القضية للمحكمة
];

// النوع 2: مدعي - قضية مقيدة (مرفوعة مسبقاً في المحكمة)
export const PlaintiffExistingStages: CaseStageValue[] = [
  "استلام",
  "استكمال_البيانات",
  "دراسة",
  "تحرير_المذكرة",         // هنا: تحرير مذكرة تكميلية أو جوابية حسب الحاجة
  "إحالة_للجنة_المراجعة",
  "الأخذ_بالملاحظات",
  "رفع_للدائرة",           // هنا: تقديم المذكرة للمحكمة (ليس رفع قضية جديدة)
];

// النوع 3: مدعى عليه
export const DefendantStages: CaseStageValue[] = [
  "استلام",
  "استكمال_البيانات",
  "دراسة",
  "تحرير_المذكرة",         // هنا: تحرير المذكرة الجوابية (الرد على الدعوى)
  "إحالة_للجنة_المراجعة",
  "الأخذ_بالملاحظات",
  "رفع_للدائرة",           // هنا: تقديم المذكرة الجوابية للمحكمة
];

// دالة لجلب مراحل القضية حسب التصنيف
export function getStagesForClassification(classification: CaseClassificationValue): CaseStageValue[] {
  switch (classification) {
    case "مدعي_قضية_جديدة": return PlaintiffNewStages;
    case "مدعي_قضية_مقيدة": return PlaintiffExistingStages;
    case "مدعى_عليه": return DefendantStages;
    default: return PlaintiffNewStages;
  }
}

// تسميات المراحل تتغير حسب التصنيف
export function getStageLabel(stage: CaseStageValue, classification: CaseClassificationValue): string {
  // المراحل المشتركة (نفس التسمية)
  const commonLabels: Partial<Record<CaseStageValue, string>> = {
    "استلام": "استلام",
    "استكمال_البيانات": "استكمال البيانات",
    "دراسة": "دراسة",
    "إحالة_للجنة_المراجعة": "إحالة للجنة المراجعة",
    "الأخذ_بالملاحظات": "الأخذ بالملاحظات",
  };

  if (commonLabels[stage]) return commonLabels[stage]!;

  // المراحل المختلفة حسب التصنيف
  if (stage === "تحرير_المذكرة") {
    switch (classification) {
      case "مدعي_قضية_جديدة": return "تحرير صحيفة الدعوى";
      case "مدعي_قضية_مقيدة": return "تحرير المذكرة";
      case "مدعى_عليه": return "تحرير المذكرة الجوابية";
    }
  }

  if (stage === "رفع_للدائرة") {
    switch (classification) {
      case "مدعي_قضية_جديدة": return "رفع الدعوى للمحكمة";
      case "مدعي_قضية_مقيدة": return "تقديم المذكرة للمحكمة";
      case "مدعى_عليه": return "تقديم المذكرة الجوابية للمحكمة";
    }
  }

  return CaseStageLabels[stage] || stage;
}
```

## القسم 3: تعديل نموذج إنشاء القضية

### في `client/src/pages/cases.tsx` - نموذج إضافة القضية:

#### 3.1 أضف حقل تصنيف القضية كأول حقل في النموذج (قبل كل شيء):

يكون على شكل 3 بطاقات (Cards) كبيرة قابلة للاختيار:

```
┌─────────────────────┐  ┌─────────────────────┐  ┌─────────────────────┐
│   ⚔️ مدعي            │  │   ⚔️ مدعي            │  │   🛡️ مدعى عليه      │
│   قضية جديدة         │  │   قضية مقيدة         │  │                     │
│                     │  │                     │  │                     │
│ نرفع القضية من      │  │ القضية مرفوعة مسبقاً  │  │ وردت لنا دعوى       │
│ الصفر في المحكمة     │  │ في المحكمة           │  │ ونحن المدعى عليهم    │
└─────────────────────┘  └─────────────────────┘  └─────────────────────┘
```

- هذا الحقل **إجباري** ويظهر أولاً
- بعد الاختيار تظهر بقية حقول النموذج
- التصنيف لا يمكن تغييره بعد إنشاء القضية (غير قابل للتعديل)

#### 3.2 تغيير الحقول الظاهرة حسب التصنيف:

**النوع 1 (مدعي - قضية جديدة):**
- كل الحقول الحالية كما هي
- حقول المحكمة (courtName, courtCaseNumber, circuitNumber, judgeName) تكون **اختيارية** (تُملأ لاحقاً بعد القيد)

**النوع 2 (مدعي - قضية مقيدة):**
- كل الحقول الحالية
- حقول المحكمة تكون **إجبارية** (لأن القضية مقيدة فعلاً)
- حقل إضافي: "رقم القضية في المحكمة" (courtCaseNumber) **إجباري**
- حقل إضافي: "عدد الجلسات السابقة" (previousHearingsCount) - رقم تقريبي
- حقل إضافي: "ملخص الوضع الحالي" (currentSituation) - textarea لشرح أين وصلت القضية

**النوع 3 (مدعى عليه):**
- كل الحقول الحالية
- حقول المحكمة **إجبارية**
- حقل "رقم القضية في المحكمة" **إجباري**
- حقل "تاريخ الجلسة القادمة" (إذا معروف) - لأن المدعى عليه غالباً يأتينا وعنده موعد جلسة
- حقل "المهلة المتاحة للرد" (responseDeadline) - كم يوم متبقي للرد
- الخصم هنا هو "المدعي" (غيّر label "بيانات الخصم" تبقى كما هي لكن السياق يتغير)

## القسم 4: تعديل شريط مراحل القضية

### في مكون CaseProgressBar:

#### 4.1 استخدم تسميات المراحل الديناميكية:
بدلاً من تسميات ثابتة، استخدم دالة `getStageLabel(stage, classification)`:

- **مدعي - قضية جديدة**: استلام → استكمال البيانات → دراسة → **تحرير صحيفة الدعوى** → لجنة المراجعة → الأخذ بالملاحظات → **رفع الدعوى للمحكمة**
- **مدعي - قضية مقيدة**: استلام → استكمال البيانات → دراسة → **تحرير المذكرة** → لجنة المراجعة → الأخذ بالملاحظات → **تقديم المذكرة للمحكمة**
- **مدعى عليه**: استلام → استكمال البيانات → دراسة → **تحرير المذكرة الجوابية** → لجنة المراجعة → الأخذ بالملاحظات → **تقديم المذكرة الجوابية للمحكمة**

#### 4.2 المرحلة الابتدائية حسب التصنيف:
عند إنشاء القضية:
- **مدعي - قضية جديدة**: تبدأ من "استلام" (المرحلة الأولى)
- **مدعي - قضية مقيدة**: تبدأ من "استلام" لكن يمكن تخطي المراحل بسرعة لأن القضية مقيدة أصلاً
- **مدعى عليه**: تبدأ من "استلام"

## القسم 5: تعديل الإنشاء التلقائي للمذكرات حسب التصنيف

### 5.1 عند إنشاء القضية:

**مدعي - قضية جديدة:**
- عند وصول القضية لمرحلة "تحرير_المذكرة": أنشئ مذكرة تلقائية من نوع `"تحرير_دعوى"` بعنوان "صحيفة دعوى - [رقم القضية]"

**مدعى عليه:**
- عند إنشاء القضية مباشرة: أنشئ مذكرة تلقائية من نوع `"مذكرة_جوابية"` بعنوان "مذكرة جوابية - [رقم القضية]"
- الموعد النهائي: إذا حدد المستخدم "المهلة المتاحة للرد" (responseDeadline) استخدمها، وإلا اجعلها 15 يوم من تاريخ الإنشاء
- الأولوية: "عاجل" دائماً لأن المدعى عليه لديه مهلة محددة

**مدعي - قضية مقيدة:**
- لا تُنشأ مذكرة تلقائياً عند الإنشاء (لأن القضية مقيدة ونحتاج أولاً فهم الوضع)
- تُنشأ المذكرات حسب الحاجة يدوياً أو بعد نتائج الجلسات

### 5.2 تعديل الإنشاء التلقائي بعد نتائج الجلسات:

القواعد الموجودة في الأمر السابق (الشامل) تبقى كما هي لأنها تعمل مع كل الأنواع:
- تأجيل مع طلب رد → مذكرة جوابية (تعمل مع كل الأنواع)
- حكم ضدنا مع اعتراض → لائحة اعتراضية (تعمل مع كل الأنواع)

**إضافة جديدة:** بعد صدور حكم ابتدائي ضدنا والاعتراض عليه وصدور حكم استئنافي ضدنا:
- أنشئ مذكرة تلقائية من نوع `"لائحة_نقض"` بعنوان "لائحة نقض - [رقم القضية]"

## القسم 11: إشعار خاص بقضايا المدعى عليه

### عند إنشاء قضية من نوع "مدعى عليه":
1. إشعار فوري **عاجل** لرئيس القسم: "وردت قضية جديدة نحن فيها مدعى عليهم - [رقم القضية] - المهلة: [المهلة]"
2. إشعار لمدير الفرع
3. إنشاء مذكرة جوابية تلقائية (كما في القسم 5)
4. إذا حُدد تاريخ جلسة قادمة: إنشاء الجلسة تلقائياً في نظام الجلسات

### تنبيهات المهلة (للمدعى عليه):
- قبل 5 أيام من انتهاء المهلة → تنبيه للمحامي المكلف
- قبل 3 أيام → تنبيه عاجل للمحامي + رئيس القسم
- قبل يوم واحد → تنبيه حرج للمحامي + رئيس القسم + مدير الفرع

## القسم 12: زر إحالة المذكرة للجنة المراجعة (تأكيد وتوضيح)

### 12.1 الزر في صفحة المذكرات وفي تفاصيل القضية:

هذا الزر هو الأهم في سير عمل المذكرة. يجب أن يكون **واضحاً وبارزاً**:

**في تفاصيل المذكرة (Dialog):**
- عندما تكون حالة المذكرة "قيد_التحرير": يظهر زر كبير وبارز بلون أخضر: **"إحالة للجنة المراجعة"**
- عند الضغط عليه:
  1. تتغير حالة المذكرة إلى "قيد_المراجعة"
  2. يُرسل إشعار فوري لـ **رئيس لجنة مراجعة القضايا** (cases_review_head) بنص: "تم إحالة مذكرة للمراجعة: [عنوان المذكرة] - القضية: [رقم القضية] - المحامي: [اسم المحامي] - الموعد النهائي: [الموعد]"
  3. يُرسل إشعار لـ **رئيس القسم** للعلم
  4. يظهر toast تأكيد: "تم إحالة المذكرة للجنة المراجعة بنجاح"

**في تبويب المذكرات داخل تفاصيل القضية:**
- بجانب كل مذكرة حالتها "قيد التحرير" يظهر زر صغير "إحالة للمراجعة"
- نفس السلوك عند الضغط

**في صفحة المذكرات الرئيسية (memos.tsx):**
- في عمود الإجراءات: زر "إحالة" يظهر فقط للمذكرات في حالة "قيد التحرير"

### 12.2 لوحة المراجعة لرئيس لجنة المراجعة:

عندما يدخل رئيس لجنة المراجعة صفحة المذكرات:
- يرى تبويب خاص **"بانتظار المراجعة"** يعرض فقط المذكرات المحالة إليه
- كل مذكرة تعرض: العنوان، القضية، المحامي، الموعد النهائي، الأولوية
- عند فتح المذكرة يرى زرين:
  - **"اعتماد"** (أخضر): يغيّر الحالة إلى "معتمدة" + إشعار للمحامي
  - **"إرجاع مع ملاحظات"** (أحمر): يفتح حقل لكتابة الملاحظات، يغيّر الحالة إلى "تحتاج_تعديل" + إشعار عاجل للمحامي مع نص الملاحظات + يزيد عدّاد returnCount

### 12.3 الإشعارات المتعلقة بالمراجعة:

## الملفات المتأثرة:
- `shared/schema.ts` - حقول + تصنيفات + دوال المراحل الديناميكية
- `server/routes.ts` - إنشاء مذكرات/جلسات تلقائية حسب التصنيف
- `server/storage.ts` - دوال التخزين المحدّثة
- `client/src/pages/cases.tsx` - نموذج الإنشاء + شريط المراحل + جدول القضايا + تفاصيل
- `client/src/lib/dashboard-context.tsx` - ويدجت جديد
- `client/src/lib/cases-context.tsx` - دوال إضافية

## ملاحظة مهمة:
هذا الأمر تكميلي ويُنفَّذ **بعد** الأمر الشامل (الجلسات + المذكرات). لا يتعارض معه بل يبني عليه ويضيف طبقة التصنيف.
